PHASE 1 Security & Correctness — Fix before any user touches it 3 done · 0 partial · 0 open ~2–3 weeks

Multi-tenancy is the entire product contract. If Company A can read Company B's data, nothing else matters — not the UI, not the feature set, not the performance. These tasks must be done before any external beta.

1.1 — Full IDOR audit: cross-tenant isolation test for every endpoint
AGENTS.md explicitly admits company_id FK is missing from some columns. Every API route and every Server Action needs a test proving a session from Company B cannot read, write, or delete a resource owned by Company A. This is the only way to know where the leaks are — and there will be leaks.
Done Large ↑ Security 9→10
Evidence: docs/idor-audit-matrix.md signed off — zero known leaks. CI runs npm run test:idor and npm run check:idor-coverage. FK migrations 0014, 0019.
Acceptance criteria
  • Every GET/POST/PATCH/DELETE route in src/app/api/ has a cross-tenant test asserting 403 or 404
  • Every Server Action in src/actions/ has a matching cross-tenant test
  • A CI job runs these tests in isolation — failure blocks merge to main
  • All missing company_id FKs identified in AGENTS.md are added to schema with a migration
  • Zero cross-tenant data leaks in the audit output
1.2 — Consolidate Server Actions vs API routes (pick one per resource)
Right now mutations exist in both src/actions/ and src/app/api/ for the same resources. A bug fixed in the Action is still live in the API route and vice versa. This doubles the IDOR surface from task 1.1, doubles the test burden forever, and makes every future feature slower to build safely. Pick the Server Action as the canonical layer (it has RBAC helpers already); keep API routes only for external/webhook consumers.
Done Large ↑ API consistency 6→9
Evidence: Duplicate CRUD REST removed. AGENTS.md + docs/rbac-audit-matrix.md document Server Actions as canonical. Kept API routes: PDF, cron, health, operator sub-routes only. PRD still marked Draft.
Acceptance criteria
  • Each resource (tickets, clients, services, users, companies) has exactly one mutation path
  • Server Actions are canonical for UI-driven mutations; API routes remain only for cron/webhook consumers
  • Duplicate routes removed; no dead code remains
  • All existing tests pass; no regression on RBAC checks
  • AGENTS.md updated to document the canonical pattern
1.3 — Move plan limits from code to DB
PLAN_LIMITS in company-entitlements.ts is hardcoded. Changing a limit requires a deploy. Every SaaS eventually needs to offer custom limits per customer, grace periods, or promotional overrides — none of which are possible without this change. Entitlement checks already exist in the codebase; this is just giving them a proper data source.
Done Medium ↑ Entitlements 6→9
Evidence: PRD #211 shipped via PRs #217–#219. Migration 0020_plan_catalog adds Plan, Company.plan_id, and entitlement overrides. Guards, entitlements API, Company form/list, and operator console now resolve effective limits from DB-backed plans; settings.plan production reads are blocked by an architecture test.
Acceptance criteria
  • New plans table: id, name, limits (jsonb), created_at
  • company table has plan_id FK replacing the JSONB settings.plan field
  • Migration backfills existing companies from settings JSON to plan FK
  • Operator console UI shows plan assignment and allows override per company
  • All entitlement checks read from DB; hardcoded PLAN_LIMITS object deleted
  • Existing entitlement tests updated and pass; architecture guard prevents legacy plan reads
PHASE 2 Architecture — Stop accruing technical debt 2 done · 2 partial · 1 open ~2 weeks

These aren't emergencies but they compound. Every week you wait, more code is written against the broken pattern, making the fix larger. The right time to do these is immediately after Phase 1, while the codebase is still manageable.

2.1 — Decompose oversized components
tickets-list.tsx (36KB) and ticket-services-list-client.tsx (32KB) are already painful to work in. At this size, every change risks unintended side effects, tests are hard to scope, and new developers can't orient quickly. The decomposition also forces clean prop/interface boundaries which improve testability.
Done Medium ↑ Architecture 7→9
Evidence: Decomposed components under 300 lines. 12 unit test suites in src/components/tickets/. Visual baselines in e2e/tickets-visual.spec.ts-snapshots/ (npm run test:e2e:visual).
Acceptance criteria
  • No single component file exceeds 300 lines / ~8KB
  • tickets-list.tsx split into: list container, row, filter bar, sort controls, mobile card
  • ticket-services-list-client.tsx split into: services table, service row, add-service panel
  • All existing tests pass; new unit tests for extracted sub-components
  • No visual regression (Playwright screenshot baseline)
2.2 — N+1 query audit and index review
Tenant-scoped list queries — tickets, clients, audit events — hit the database with a WHERE company_id = ? filter on every request. Without proper indexes and a review of relation loading, these queries will degrade linearly as data grows. A company with 12 months of ticket history should not feel slower than a new company.
Done Medium ↑ Performance baseline
Done: Soft-delete-aware list indexes in migration 0023; docs/query-budget.md; npm run seed:perf + npm run query:audit; 10k/1k baseline recorded 2026-07-24.
Acceptance criteria
  • EXPLAIN ANALYZE run on all list queries for Ticket, Client, Service, AuditEvent, User
  • All sequential scans on tenant-scoped queries replaced with index scans
  • No N+1 patterns in list endpoints — relations eager-loaded in single query
  • Query budget documented: acceptable ms thresholds per endpoint in docs/query-budget.md
  • Seed dataset of 10k tickets / 1k clients used as perf baseline, results recorded
2.3 — Expand RBAC test coverage to full role matrix
rbac-role-matrix.test.ts exists but the current coverage is incomplete. The permissions system is custom-built — no external library enforcing it. Every permission (tickets:create, clients:delete, etc.) needs a test asserting it blocks the right roles and allows the right roles. Gaps here cause silent privilege escalation.
Partial Medium ↑ RBAC 8→10
Evidence: rbac-role-matrix.test.ts, rbac-coverage.test.ts, docs/rbac-audit-matrix.md, RBAC epic shipped. Missing full permission×role matrix, >90% coverage gate on *-rbac*.ts, CI fail on new permission without test.
Acceptance criteria
  • Every permission in error-catalog.ts has at least one allow and one deny test — permissions tested via RBAC helpers, not full matrix
  • Role matrix table in tests: each role × each permission = expected result documented
  • System user bypass tested explicitly: super-admin can access all, regular user cannot
  • Coverage report shows >90% on all files under src/lib/*-rbac*.ts
  • CI fails if a new permission is added without a corresponding test
2.4 — Error handling audit: unify action and API error shapes
Server Actions return { success, data?, error?, errorType? } while API routes use ok() / fail() helpers with different shapes. The frontend currently handles both inconsistently. Unifying these means error handling can be written once, toast messages are consistent, and the error-catalog.ts codes are actually surfaced to users rather than lost.
Partial Small ↑ Error handling 7→9
Evidence: error-catalog.ts (55 codes), handleServerActionError, ok()/fail(), getErrorMessageByType. Missing shared ActionResult<T> / ApiResponse<T> types, consistent Sentry context, unified component error flow.
Acceptance criteria
  • Single ActionResult<T> type used by all Server Actions — ad-hoc inline shapes instead
  • Single ApiResponse<T> type used by all API routes — ok()/fail() helpers exist, no shared type
  • All 55 error codes in error-catalog.ts map to a user-visible toast message
  • No raw try/catch blocks in components — all errors flow through shared handler — mixed patterns remain
  • Sentry receives structured error context on every failure (error code + resource type) — Sentry wired, context not uniform
PHASE 3 Quality & Observability — Know when things break 2 done · 1 partial · 0 open ~1–2 weeks

Once the architecture is clean and security is tight, you need the instrumentation to run this in production with confidence. These tasks close the gaps in test coverage, observability, and API consistency.

3.1 — Increase test coverage on critical paths
75 test files for 399 source files (18.7%) is thin for a security-critical multi-tenant app. The priority is not coverage percentage as a metric — it's covering the paths where a bug causes data loss or security failure: auth flows, RBAC checks, ticket financials, audit event writes, and the job queue retry logic.
Partial Large ↑ Test coverage 7→9
Evidence: CI coverage gate at 25% global (jest.config.js), not 70%. ticket-financials.test.ts and money.test.ts exist. Job queue: backoff math only. No auth/token_version or audit-outbox tests.
Acceptance criteria
  • Auth flow (login, 2FA, token version invalidation) at 100% branch coverage — 2FA E2E exists, no token_version unit tests
  • Ticket financials (ticket-financials.ts, money.ts) at 100% — tests exist, not full branch coverage
  • Audit event writes covered: success path, failure path, outbox fallback
  • Job queue: enqueue, claim, complete, retry backoff, exhaustion all tested — backoff only
  • Overall coverage gate of 70% added to CI — build fails below threshold — gate exists at 25%
3.2 — Structured logging with request correlation IDs
logger.ts exists and Sentry is wired up, but without request-level correlation IDs it's nearly impossible to trace a user complaint ("my ticket save failed") to a specific log line in production. A correlation ID threaded through every log call turns a 2-hour debug session into a 2-minute log query.
Done Small ↑ Observability 7→9
Evidence: src/lib/request-context.ts + ALS bind in requireActionAuth/requireSession; src/proxy.ts propagates x-request-id for pages and /api/*; health echoes id; docs/observability.md. Shipped Jul 2026.
Acceptance criteria
  • Every incoming request gets a x-request-id header (generated if absent)
  • Request ID threaded through all logger calls in API routes and Server Actions
  • Sentry captureException calls include requestId in context
  • /api/health returns request ID in response for smoke-test traceability
  • Log format documented in docs/observability.md
3.3 — Onboarding checklist for new companies
New Company tenants need a structured activation path on the dashboard. The in-app Inicio rápido checklist mirrors guia-empresa.html, auto-detects progress from tenant data, links each step to create flows and guide anchors, and supports admin dismiss persisted in Company.settings.
Done Medium ↑ Activation UX
Evidence: company-onboarding-checklist.ts, fetchOnboardingStatus / dismissOnboardingChecklist, DashboardOnboardingHelp (6 steps, progress, RBAC-gated actions, guide deep links). Shipped PRD #220 → main Jul 2026.
Acceptance criteria
  • Checklist shows on dashboard when company has fewer than 3 of: services, clients, tickets, or any step remains incomplete
  • Steps: Mi empresa → servicios → clientes → primer ticket → invitar equipo → factura PDF y recordatorios (guide order + team step)
  • Each step links directly to the relevant create flow
  • Checklist auto-hides once all six steps are complete
  • Progress persists across sessions (DB-derived); dismissable by admin via onboarding_checklist_dismissed_at
PHASE 4 Polish — Close the remaining partial scores 1 done · 1 partial · 1 open ~1 week

These are the remaining 7/10 items — good but not great. They're last because they don't block shipping, but they matter for the product feeling like a professional, complete tool.

4.1 — PWA offline detection and graceful degradation
The app is installable but has no offline mode and no graceful offline message. A user who installs the PWA and loses network sees a blank screen with no explanation. The README acknowledges this as a v1 limitation. Add a simple offline banner and cache the shell so installed users see a clear "you're offline" state instead of a broken app.
Done Small ↑ PWA 7→9
Evidence: @serwist/turbopack SW at /serwist/sw.js, src/app/sw.ts, /offline fallback, AppSerwistProvider (prod-only), NetworkStatusBanner, e2e/mobile-offline.spec.ts. Shipped PRD #228 → main Jul 2026.
Acceptance criteria
  • Service worker caches app shell (HTML, CSS, JS) on install
  • Offline banner renders immediately when network is lost — no blank screen
  • Banner auto-dismisses when connection is restored
  • Cached shell allows navigating between already-loaded pages while offline
  • Playwright mobile test: go offline → assert banner visible → come back online → assert banner gone
4.2 — Feature flags system (lightweight, DB-backed)
No feature flags means every experiment, gradual rollout, or per-company beta feature requires a code deploy or a hardcoded env var. As soon as there are multiple customers, you'll need to give one company early access to something without giving it to everyone. A simple DB-backed flags table is enough — no need for a third-party service at this stage.
Not started Medium ↑ Operational flexibility
Gap: PostHog wrapper only (src/lib/analytics.ts, src/lib/flags.ts). No feature_flags DB table, operator UI, or 60s TTL cache.
Acceptance criteria
  • New feature_flags table: key, enabled (bool), company_ids (int[] nullable)
  • isFeatureEnabled(key, companyId) helper — global if company_ids is null, scoped if set
  • Flags readable in Server Actions, API routes, and React Server Components
  • Operator console UI to toggle flags per company without a deploy
  • Cached with 60s TTL to avoid a DB call on every render
4.3 — Mobile E2E coverage for core flows
Playwright mobile-chrome tests exist for navigation and layout, but the core business flows — create ticket, add service to ticket, collect payment — are only tested on desktop. Given the mobile-first initiative tracked in tasks/INDEX.md and the PWA push, these flows need mobile E2E coverage before they can be considered production-ready on phone.
Partial Medium ↑ Mobile coverage 7→9
Evidence: mobile-chrome in CI. Mobile specs cover layout/navigation (tickets-mobile.spec.ts, sidebar, sticky nav). Full business flow in e2e/core-flow.spec.ts runs desktop-only (1280×900).
Acceptance criteria
  • Mobile E2E test: create ticket end-to-end on Pixel 5 profile — create flow layout only, not full E2E
  • Mobile E2E test: add service to ticket, verify total updates correctly
  • Mobile E2E test: collect payment on ticket, verify status changes to paid
  • Mobile E2E test: PDF invoice download works on mobile Chrome
  • All tests run in CI on mobile-chrome Playwright project — failures block merge
What this plan does not include: Stripe, subscription lifecycle, transactional email, password reset flow, self-serve signup — those are a separate workstream. Complete this plan first; the commercial layer lands on top of a trustworthy foundation.
