## What to build

End-to-end foundation for **Client service schedules**: Drizzle schema and migration for `ClientServiceSchedule`, pure modules for calendar-month/day interval math and due-state bucketing (Próximos 14d, Atrasados, Programados, Pausados), and Server Actions to list (with filter), upsert one Client+Service pair, pause, resume, and soft-delete—scoped by Company, respecting soft-deleted Client/Service joins, and RBAC (`tickets.read` to list; `tickets.write` or `clients.write` to mutate).

When a Client or Service is soft-deleted via existing actions, related schedules are auto-paused so reminders do not reference deleted catalog rows.

Verified via unit tests on pure modules and action-level tests or integration tests where the repo already patterns them.

## Acceptance criteria

- [ ] Migration creates `ClientServiceSchedule` with unique active `(company_id, client_id, service_id)`, interval fields, `last_service_at`, `next_due_at`, `paused_at`, optional `pause_reason`, soft-delete
- [ ] Pure date math: calendar months with end-of-month clamp; days; `next_due` from `last_service_at` + interval
- [ ] Pure bucketing: Próximos (≤14 days), Atrasados, Programados, Pausados; paused excluded from urgent buckets
- [ ] Server Actions: list with filter + company scope; upsert; pause/resume; soft-delete; permission checks
- [ ] Soft-delete Client or Service pauses related schedules (no auto-unpause on restore)
- [ ] `npm test`, `npm run lint`, `npm run build` pass
